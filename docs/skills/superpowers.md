# Superpowers

> 一套完整的 AI 编码工作流框架 — 让 AI 不再"随便写写"，而是遵循严格的软件工程纪律来开发。

## 这是什么？

Superpowers 是目前最受欢迎的 Claude Code 插件之一（GitHub 72k+ stars）。它不是一个简单的工具，而是一整套 **开发工作流规范**，由 15+ 个 skill 组成。

简单来说：**装了 Superpowers 之后，AI 会变得更靠谱。**

没装之前，AI 可能会：
- 想到什么写什么，没有计划
- 跳过测试，直接写代码
- 遇到 bug 乱猜，反复试错
- 写完就说"搞定了"，实际上没验证过

装了之后，AI 会被强制要求：
- 先头脑风暴，再做设计，最后才写代码
- 严格遵循 TDD（先写测试，再写实现）
- 遇到 bug 按科学方法排查，不许乱猜
- 完成前必须拿出验证证据，不许说"应该没问题"

## 核心理念

Superpowers 的设计基于四个原则：

1. **测试驱动开发（TDD）** — 先写测试，再写代码，不许反过来
2. **系统化流程优于猜测** — 遇到问题按步骤排查，不许"试试看"
3. **简单至上** — 不写不需要的代码（YAGNI），不过度工程化
4. **用证据说话** — 说"通过了"就必须拿出测试结果，不许凭感觉

## 工作流全景

Superpowers 把一个完整的开发任务拆成了这样的流程：

```
头脑风暴 → 写计划 → 建隔离分支 → 按计划执行 → 代码审查 → 合并/PR
```

每一步都有对应的 skill 在背后控制 AI 的行为。下面逐一介绍。

---

## Skill 详解

### 1. Brainstorming（头脑风暴）

**触发时机**：当你说"我想做一个 XX 功能"时自动触发。

**它做什么**：

AI 不会直接开始写代码。它会先：

1. **了解项目现状** — 读代码、看文档、查最近的提交
2. **逐个提问澄清** — 每次只问一个问题，搞清楚你到底要什么
3. **提出 2-3 种方案** — 带上各自的优缺点和推荐意见
4. **分段展示设计** — 每展示一部分就问你"这样可以吗？"
5. **写设计文档** — 保存到 `docs/plans/` 目录
6. **交给下一步** — 自动进入计划编写阶段

**关键规则**：

> 设计没批准之前，不写任何一行代码。不管多简单的任务都是。

哪怕你觉得"就改一行代码"，AI 也会先确认你要什么再动手。这是故意设计的 — 因为"简单"任务往往是假设最多、翻车最多的地方。

---

### 2. Writing Plans（写计划）

**触发时机**：头脑风暴完成、设计获批后自动触发。

**它做什么**：

把设计方案拆成非常细的执行步骤。有多细？每一步只需要 2-5 分钟：

```markdown
### 任务 1：用户验证函数

**步骤 1：写失败测试**
（具体的测试代码）

**步骤 2：运行测试，确认失败**
运行命令：pytest tests/test_auth.py -v
预期结果：FAIL

**步骤 3：写最小实现**
（具体的实现代码）

**步骤 4：运行测试，确认通过**
运行命令：pytest tests/test_auth.py -v
预期结果：PASS

**步骤 5：提交**
git commit -m "feat: add user validation"
```

每个步骤都包含：
- 精确的文件路径
- 完整的代码（不是"在这里加验证逻辑"这种模糊描述）
- 具体的运行命令和预期输出

**写完后提供两种执行方式**：
- **子代理模式**（当前会话）— 每个任务派一个独立的 AI 去做
- **新会话模式** — 开一个新对话来执行

---

### 3. Test-Driven Development（测试驱动开发）

**触发时机**：写任何新功能或修 bug 时自动触发。

**它做什么**：

强制 AI 遵循 TDD 的 Red-Green-Refactor 循环：

```
RED：写一个会失败的测试
     ↓
运行测试，确认它确实失败了
     ↓
GREEN：写最少的代码让测试通过
     ↓
运行测试，确认通过了
     ↓
REFACTOR：清理代码
     ↓
回到 RED，写下一个测试
```

**铁律**：

> 没有失败的测试，就不许写生产代码。先写了代码？删掉，从头来。

这不是建议，是强制规则。AI 如果先写了代码再补测试，这个 skill 会要求它把代码删掉重来。

**为什么这么严格？**

先写代码再补测试，会导致：
- 测试一跑就过 → 证明不了测试是有效的
- 你只会测"代码做了什么" → 而不是"代码该做什么"
- 遗漏边界情况 → 因为写代码时没想到

先写测试再写代码：
- 看到测试先失败 → 证明测试确实在检测这个功能
- 强制你思考"代码该做什么" → 而不是凑实现
- 逼你发现边界情况 → 因为你在写期望行为，不是凑代码

**常见借口与反驳**：

| 借口 | 现实 |
|------|------|
| "太简单了，不用测试" | 简单代码也会出 bug，测试只要 30 秒 |
| "我先写完再补测试" | 后补的测试一跑就过，啥也证明不了 |
| "TDD 太慢了" | TDD 比"写完 debug 半天"快多了 |
| "我已经手动测过了" | 手动测试没法重复运行，下次改代码还得再测 |
| "删掉 X 小时的代码太浪费了" | 沉没成本。留着没测试的代码才是真浪费 |

---

### 4. Systematic Debugging（系统化调试）

**触发时机**：遇到任何 bug、测试失败、异常行为时自动触发。

**它做什么**：

禁止 AI 猜测性修复。强制按四个阶段排查：

**阶段 1：找根因**
- 完整阅读错误信息（不许跳过）
- 稳定复现问题
- 检查最近的代码变更
- 追踪数据流，找到出错的源头

**阶段 2：模式分析**
- 找到类似的能正常工作的代码
- 对比正常和异常的区别
- 列出所有差异（再小也不能忽略）

**阶段 3：假设与验证**
- 提出一个明确的假设："我认为 X 是根因，因为 Y"
- 做最小改动来验证假设
- 一次只改一个变量

**阶段 4：实施修复**
- 先写一个复现 bug 的测试
- 修复根因（不是症状）
- 确认测试通过

**关键规则**：

> 如果连续 3 次修复都没用，停下来 — 这不是代码问题，是架构问题。

AI 不会无限次"再试一下"。三次失败后会停下来和你讨论是否需要重新设计。

**常见借口与反驳**：

| 借口 | 现实 |
|------|------|
| "问题很简单，不用走流程" | 简单问题也有根因，流程对简单 bug 很快的 |
| "紧急情况，没时间排查" | 系统排查比反复猜测快得多 |
| "我觉得是 X，先改了试试" | 看到症状 ≠ 理解根因 |

---

### 5. Verification Before Completion（完成前验证）

**触发时机**：AI 打算说"搞定了""应该没问题了"之前自动触发。

**它做什么**：

禁止 AI 在没有证据的情况下声称完成。

**铁律**：

> 如果没有在当前消息中运行验证命令，就不许说"通过了"。

具体来说：

```
❌ "测试应该通过了"
❌ "看起来没问题"
❌ "我很有信心"

✅ [运行 npm test] [看到：34/34 通过] "所有测试通过"
✅ [运行 build 命令] [看到：exit 0] "构建成功"
```

**这意味着什么？**

AI 每次声称"完成"之前，都必须：
1. 确定用什么命令验证
2. 真正运行这个命令
3. 读完整的输出
4. 输出确认了才能说

不是"应该"，不是"大概"，不是"我很有信心"。必须有证据。

---

### 6. Using Git Worktrees（Git 工作树）

**触发时机**：开始新功能开发时自动触发。

**它做什么**：

在开始写代码之前，先创建一个隔离的工作空间：

1. 检查项目是否已有 worktree 目录
2. 创建新分支和工作树
3. 安装依赖
4. 运行测试确认基线正常
5. 报告就绪状态

**为什么要这样？**

在隔离的分支上工作意味着：
- 不会污染主分支
- 随时可以丢弃重来
- 多个功能可以并行开发

---

### 7. Executing Plans（执行计划）

**触发时机**：在新会话中执行之前生成的计划时使用。

**它做什么**：

加载计划文件，按批次执行（默认每批 3 个任务）：

1. 读取并审查计划（有疑问先提出来）
2. 执行一批任务
3. 展示完成情况，等待反馈
4. 根据反馈调整，执行下一批
5. 全部完成后进入合并流程

**关键规则**：遇到阻塞（依赖缺失、测试失败、指令不清）立即停下来问你，不会硬猜。

---

### 8. Subagent-Driven Development（子代理驱动开发）

**触发时机**：在当前会话中执行计划时使用（与 Executing Plans 的区别是不需要开新会话）。

**它做什么**：

每个任务派一个全新的 AI 子代理去执行，然后做双重审查：

```
任务 N →
  派子代理实现 →
    子代理实现完成 →
      第 1 轮审查：符合规格吗？ →
        不符合 → 子代理修复 → 再审查
        符合 →
      第 2 轮审查：代码质量好吗？ →
        不好 → 子代理修复 → 再审查
        好 → 标记完成 → 下一个任务
```

**为什么每个任务用新的子代理？**

- 新代理没有之前的上下文污染
- 每个任务的范围清晰隔离
- 子代理可以先提问再动手
- 双重审查（规格合规 + 代码质量）保证输出质量

---

### 9. Dispatching Parallel Agents（并行代理调度）

**触发时机**：多个独立的问题需要同时处理时使用。

**它做什么**：

当出现多个互不相关的问题时（比如 3 个不同文件的测试失败），不是一个一个排查，而是同时派 3 个代理并行处理。

**什么时候该用**：
- 3+ 个测试文件失败，原因各不相同
- 多个子系统独立出问题
- 各问题之间没有共享状态

**什么时候不该用**：
- 问题之间有关联（修一个可能修好其他的）
- 代理会互相冲突（比如改同一个文件）

---

### 10. Requesting / Receiving Code Review（代码审查）

**Requesting（请求审查）**：

每个任务完成后自动派审查子代理，检查代码的：
- 规格合规性（功能对不对）
- 代码质量（写得好不好）

**Receiving（接收审查反馈）**：

收到审查意见后的处理规则：
- 不许说"你说得对！"（这是敷衍）
- 先验证建议是否正确
- 如果建议不对，用技术理由反驳
- 如果建议正确，直接修，不用感谢

**处理优先级**：
1. 阻塞性问题 → 立即修
2. 重要问题 → 继续之前必须修
3. 次要问题 → 记下来以后处理

---

### 11. Finishing a Development Branch（完成开发分支）

**触发时机**：所有任务执行完毕后自动触发。

**它做什么**：

1. 先运行测试确认全部通过
2. 提供 4 个选项让你选：
   - 合并回主分支
   - 推送并创建 PR
   - 保持现状（你自己处理）
   - 丢弃这些改动
3. 按你的选择执行
4. 清理工作树

---

### 12. Writing Skills（编写 Skill）

**用途**：当你想创建自己的 skill 时使用。

这个 skill 本身就遵循 TDD — 先写"压力测试"（AI 没有这个 skill 时会怎么做），看它失败，再写 skill 内容让它通过。

## 完整工作流示例

假设你想给项目加一个用户认证功能：

```
你：我想加一个用户登录功能

[Brainstorming 触发]
AI：我先看看项目结构...你用的是什么技术栈？
你：Next.js + Prisma
AI：好的，认证方案有 3 种选择：
    1. NextAuth.js（推荐，和 Next.js 集成最好）
    2. 自己写 JWT
    3. Firebase Auth
    你倾向哪种？
你：用 NextAuth
AI：（展示设计方案，逐段确认）

[Writing Plans 触发]
AI：计划已生成，共 5 个任务，保存在 docs/plans/2026-03-06-auth.md
    1. 安装和配置 NextAuth
    2. 创建登录页面
    3. 添加 session 管理
    4. 保护需要登录的路由
    5. 添加注册流程
    用子代理模式还是新会话模式？
你：子代理

[Subagent-Driven Development 触发]
[Git Worktree 创建隔离分支]

任务 1：
  [子代理按 TDD 实现]
  [规格审查通过]
  [代码质量审查通过]
  ✅ 完成

任务 2-5：同上流程...

[所有任务完成]
[Finishing Branch 触发]
AI：全部完成，测试通过（23/23）。选择：
    1. 合并回 main
    2. 推送并创建 PR
    3. 保持现状
    4. 丢弃
你：2
AI：PR 已创建：https://github.com/...
```

## 与 yy-spec 的区别

| 维度 | Superpowers | yy-spec |
|------|-------------|---------|
| 定位 | 编码纪律框架 | 规格驱动开发工具 |
| 核心关注 | 怎么写代码（TDD、调试、审查） | 怎么管需求（需求 → 设计 → 任务 → 实现） |
| 文档产出 | 设计文档 + 执行计划 | 需求文档 + 设计文档 + 任务清单 |
| 适合场景 | 任何编码任务 | 功能开发、Bug 修复、问题调查 |
| 可以一起用 | 是 | 是 |

两者不冲突，可以同时安装。Superpowers 管"怎么写好代码"，yy-spec 管"怎么把需求理清楚"。
