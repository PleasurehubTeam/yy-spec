# 规格驱动开发（SDD）

> 30 秒了解：SDD 的核心理念是"先明确规格，再动手编码"— 规格文档是 AI 和人类共同的真相来源。

## 核心理念

传统开发中，需求常常只存在于口头讨论或简单的 ticket 描述中。SDD 要求：

1. **每个工作项都有规格** — 功能、Bug、调查，都有对应的规格目录
2. **规格文档是唯一真相** — AI 和人类都从规格中获取上下文
3. **规格可追溯** — 从需求到设计到任务到代码，每一步都有迹可循

## 规格目录结构

每个工作项在 `.yy-dev/specs/` 下有独立目录：

```
.yy-dev/specs/user-auth/
├── spec.json          # 元数据和状态
├── requirements.md    # 需求文档
├── design.md          # 技术设计
├── tasks.md           # 实现任务
└── research.md        # 研究和决策记录
```

## Steering vs Specification

| | Steering | Specification |
|---|---------|---------------|
| 位置 | `.yy-dev/steering/` | `.yy-dev/specs/<name>/` |
| 范围 | 整个项目 | 单个工作项 |
| 内容 | 项目上下文和约束 | 具体功能的需求和设计 |
| 更新频率 | 偶尔（架构变化时） | 每个工作项一次 |

## 为什么需要 SDD？

- **减少 AI 幻觉** — 明确的规格约束 AI 的输出范围
- **提高一致性** — 多次对话中 AI 都能从规格获取正确上下文
- **便于审核** — 每个阶段都有可审核的文档产出
- **知识沉淀** — 决策记录不会随对话消失
